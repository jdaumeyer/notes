Terminology

It is easy to makes tests against a reference of a method that we know works. Although this isn’t always possible if we don’t have a working reference.

UUT
Unit Under Test is the class whose implementation we are testing.
RU
Reference Unit is an implementation that we know already works.
Black Box Testing
These are test cases developed without any knowledge of the specific implementation. These are test made off of the method contracts.
  • These tests are usually just the boundary and routine cases.
White Box Testing
These are tests written with knowledge of how the UUTs were implemented. Meaning that we can write tests intended to break them.
  • You want to test every flow of control meaning going down every if statement or conditional code.

Testing

The way we will create and use test cases in this class follows a pretty structured format. The goal of these test is to make sure our implementation works exactly as expected. Meaning that there are no side effects or unintended consequences. The process goes as follows:

  1. Set up the Unit Under Test (UUT)
  2. Set up the Reference Unit (RU)
  3. Perform the same operation on both objects
  4. Test for equality
    • If the operation is a function, test return values for equality
    • if any parameter was mutable, test it for equality
    • If the objects were mutable, test them for equality

Making Good Tests

To make an ideal test, we want to create all the setup with our helper methods and then make a single call to the method being tested

Sometimes testing in isolation is not always accurate. You may need to test how your methods interact with others, these would fit into the White Box test plan.

Debugging

Testing and Debugging are different.

  • Testing is the process of finding bugs
  • Debugging is the process of fixing bugs

The Seven Steps of Debugging (TM)

Does an assertion show us where the error is?

Identify the simplest test case that fails

Reduce that case to a minimal case

Determine the last location in the test case where the representation is still correct

Use breakpoints to jump into kernel methods

Find the first location the trace doesn’t match

Fix the first error you find then return to tests
