Contracts

Although contracts follow a syntax similar to code, they don’t follow the same syntax as Java code does. While there are a lot of similarities, there are also a lot of differences.

  • Contracts are designed to describe what a program does whereas Java code worries about how it does it.
  • Contract Specifications follows a more mathematical syntax than a normal programming language.

The Contract Language

Mathematical Types

Similar to Java the Contract Specification has types, however these are all based on actual mathematical types. Java Types and Mathematical Types are NOT the same.

Boolean
either true or false
integer
a positive or negative whole number
real
represents a real number (not used in Software 2).
character
a single Unicode character.
set
an unordered collection of items that are of the same type (sets have no order and don’t maintain duplicates).
  • Represented with curlybraces \({a, b, ...}\).
string
a totally-ordered collection of items that are the same type.
  • Represented with angle brackets $<a, b, …$.
  • The concatenation operator is \(*\).
  • The entries of a string are all the unique bits taken as a set
tuple
a group of different things that can be distinguished.
  • Use parentheses \((a, b, ..)\).

Conditions

Preconditions

A precondition is simply what the method expects to receive in terms of parameters and other values. In other words, these are things that the client has to ensure are true.

  • If the precondition is not fulfilled, the behavior of the method is no longer garunteed.
  • The precondition is seen in the contract with the @requires tag.

Postconditions

A postcondition is what can be expected from the output of the method. It is the implementers responsibility to ensure this is true if the precondition is fulfilled.

  • The postcondition is seen is the method contract with the @ensures tag.

Client V. Implementer

Clients and Implementers are not seperate people but rather different perspectives on the code.

Quantifiers

The list of variables that follow a quantifier word have nothing to do with the value declarations that follow them.

\(for all\)
True iff main clause holds for all the cases.
\(there exists\)
True iff main
(no term)
When the name of a method is in the specification, it is referring to the return value of the method.
  • removeAny is in #this#, the removeAny refers to the return of the method.
