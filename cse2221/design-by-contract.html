System Thinking

System

A system is any part of anything that you want to think about as an individual unit.

Interface

An Interface is a description of the “boundary” between a system and everything else.

Subsystem (Component)

A subsystem is a system that is used inside another system.

Client V. Implementer

A client is a person (or role played by some person) viewing a system “from the outside”. These are agents that use a system.

An implementer is a person (or role played by some agent) viewing a system “from the inside”. These are the agents that make/design a system.

An agent can be both a client and an implementer. For example, when you write a method that you later use, you are both the client and implementer of that method.

Describing Behavior

Information Hiding

Information hiding is a technique for describing system information while intenionally leaving out more specific details.

Abstraction

Abstraction is a consequence of Information Hiding where we need to create a valid cover story in order to counteract the effects of hiding some details.

  • This is because the hidden information is still important so its presence still needs to be accounted for.

Method Contract

Structure

Each method has a precondition ( requires clause ) that characterizes the responsibility of the progam that calls that method, and a postcondition (ensures clause) that characterizes the responsibility of the program that implements that method.

Responsibilities and Rewards

The implementer and client both have responsibilites that the must fufill in this model, and for completing these requirements they also receive a reward. For the Method Contract, this responsibilites and rewards are:

  • The clients responsibility is to make sure the precondition is true and their reward is that they may assume the postcondition is true when the method returns
  • The implementer’s responsibility is to make sure the postcondition is true, and for this their reward is that they may assume the precondition is true.

Javadoc

Javadocs are comments used by Java to make your program more client accessible and document your work. Javadocs are separated into different tags that specify different aspects of the method.

@param
defines the parameters of the method, one is needed for each parameter
@return
specifies what the method returns.
@requires
This defines the preconditions of the method. Essentially what it expects its parameters to be.
@ensures
This defines the postcondition of the method, or what the client can expect the return value to be.
