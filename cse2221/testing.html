Testing
CSE 2221

Testing is a very common software engineering activity and is critical to ensure proper program execution. Because of this, there are many different testing methodologies. The one we will be covering in this class is Unit Testing.

Unit Testing

The process of Unit Testing involves testing an individual unit of code at a time. A unit can be something like a method or a class.

  • Testing what happens when multiple units are put together is called Integration Testing
  • Testing a entire end-user application is called System Testing

Determining Correctness

To test a unit means that we must determine whether what it’s doing it correct or not. We can actually determine the correctness by looking at the [Contract][]]. If the method fulfills its contract then it is correct.

It is the job of a test to prove that something doesn’t work, not why is doesn’t work. Determining the why is the job of Debugging.

Test Cases

The Test Cases of an application need to be very deliberately created. These will typically follow a Test Plan.

The best Practices for making a test plan include:

  • Testing boundary cases, or the smallest, largest and special cases.
  • Testing routine cases
  • Testing challenging cases

JUnit

What we have been doing up until this point is creating test code and running the program with it. This is a very primitive method of testing and not recommended for actual testing. The best practice for Java is using JUnit.

@Test Methods

You can create test methods easily by writing whatever testing that you would have in the main method into its own method and prefixing it with @Test. 

@Test
public void testMethod() {}

These methods will only run when we test our project.

The assert Keyword

assert is used throughout our Java components as a part of the method contracts. However they are technically also a test feature. Just like a normal test, they ensure that we are only using our code as intended.

  • A lot of the errors you’ll have in the software sequence are a result of violating method contracts. By having the assertions, we can catch the errors easier that having the decipher undefined behavior.
