Critical Section Problem

A Critical section is book-ended by both an entry section and an exit section.

Entry Section Manages requests to enter the critical section
Critical Section A segment of code where processes can effect common variables
Exit Section Signals that the critical section is free
Remainder Section The rest of the code

All process have some critical section or a shared resource. When one process is accessing a critical section no other processes should be allowed access at the same time. The problem is to design a protocol to ensure this.

A solution must include:

Mutual Exclusion
No more than one process is allowed access to a critical section at once
Progress
Not all processes are waiting, at least one is progressing (i.e. no deadlock)
Bounded Waiting
No processes are starving

Peterson’s Solution

Wont work on modern systems because of Out of Order Execution, but still a great tool for explaining.

  • Assume only 2 processes (Process i and process j)
  • Assume load and store instructions cannot be interrupted
  • Assume two shared variables int turn; (which indicates who is allowed to enter the critical section) and boolean flag[2]; (indicates which process is ready to enter the critical section)
while (true) {
    /* === Entry Section === */
    flag[i] = true; /* This process is ready */
    turn = j; /* Let the other process go first */
    while (flag[j] && turn == j); /* Busy Loop
                                     basically do nothing is the other process
                                     is ready and it's currently their turn */

    /* If we get here, it's our turn and we are ready */
    /* === Critical Section === */

    /* === Exit Section === */
    flag[i] = false;

    /* === Remainder Section === */

}

Tools

Atomics

Many computer systems have atomic instructions, meaning that they are uninterruptible until they finish execution. These can provide a solution to the critical section problem, using them to create locks which allow the other tools to work.

test_and_set(boolean *target)
return the value of target and then update it to true.
compare_and_swap(int *value, int expected, int new_value)
if value is equal to expected then value will be set to new_value. The original value is always returned.

Mutex Locks

Mutex Locks (short for Mut ual Ex clusion Locks) involve passing around a lock that allows access into the critical section.

Acquire Lock
Wait until the lock is available
Release Lock
Set the lock to available

Semaphores

A semaphore is an integer that is only accessible from the atomic operations wait() and signal()

  • wait() and signal() we both called P() and V() respectively from the Dutch words proberen (“to test”) and verhogen (“to increment”)
  • When the value of the semaphore is less than or equal to 0, all resources are being used. When signal() is called, it increments the value of the semaphore thus freeing a resource.

Binary Semaphore

A Binary Semaphore is essentially a mutex lock since it only allows for the values 0 and 1.

Counting Semaphore

A counting semaphore is useful for a resource that will be accessed by a finite number of instances. The value of the semaphore will be set to the number of resources available.

Bounded Buffer

We have a list of buffers that can only hold 1 item each. Some processes want to read from these buffers while others are writing to them. We want to ensure that every reader only reads from a buffer that has been written to, however this needs to work concurrently. For the Bounded Buffer problem, we have two types of processes:

Consumers
processes who read from the buffers
Producers

processes who write to the buffers

They both have the following shared information: 

int n;
semaphore mutex = 1;
semaphore empty = n;
semaphore full = 0;
(no term)
Where n is the number of buffers.
(no term)
mutex ensures that only one process has access to the buffers at a time
(no term)
empty tracks the number of empty buffers
(no term)
full tracks the number of full buffers

Readers Writers

We have a database and two types of processes that want to read from the database:

Readers
only read data from the database
Writers
update data on the database both reading and writing

Now multiple readers having access to the database at a time is fine, the problem comes in when multiple writers have access. There are two ways to solve this problem, one where readers have priority and another where writers have priority instead.
